# Group C: Report Page + User Profiles — Design Spec

## Context

Extends the Life Change Planner. Two features: a shareable report page for presenting results, and user profiles so multiple people (Mekoce, Kara, etc.) can independently configure preferences while sharing the same financial assumptions.

**Stack**: Same — Vite + React + TypeScript. No new dependencies.

**Depends on**: Groups A and B complete.

---

## Feature 1: Shareable Report Page

### Route

`#/report` — new top-level route. Add "Report" to the nav bar (between Plan and Settings).

### Content (top to bottom)

1. **Header**: "Life Change Planner — Report for [Profile Name]" + date generated
2. **Assumptions summary**: Compact block showing household income, home value, mortgage, retirement age, investment return rate, inflation rate, move year, daughter's age. One line each, two columns.
3. **Ranked destinations table**: All destinations sorted by composite score. Columns: rank, flag+name, net worth at 62, QoL score, composite score, career preset name, one-sentence narrative. Winner row highlighted.
4. **Top 3 spotlight**: Expanded cards for top 3 destinations, each with:
   - Destination name + flag + accent color
   - Key metrics: net worth at 62, vs DC delta, annual income year 1, annual expenses year 1, tax rate
   - Mini wealth chart (small, ~120px tall, with DC ghost line)
   - Pros/cons lists (from destination data)
5. **Matrix summary**: Active weight configuration (preset name or "Custom"), weight values listed compactly. Horizontal bar chart of composite scores for all destinations.
6. **Sensitivity findings**: From the matrix — which weight changes would flip the #1 and #2 ranking.
7. **DC home decision**: Which option is selected for the #1 destination, and its financial impact (sale proceeds, or rental income, or carrying cost).
8. **Move timeline**: For the #1 ranked destination — move year, return year (if set), key education milestones.
9. **Footer**: "Generated by [Profile Name] on [date]. These are planning estimates, not financial advice."

### Interactions

- "Print" button — calls `window.print()`
- Profile selector dropdown on the report page — switch to see report from another user's perspective
- No sliders, toggles, or editing
- URL: `#/report` (uses active profile)

### Styling

- Same design system (fonts, colors, variables)
- Wider reading layout — max-width 900px, generous padding
- Cards simplified — fewer borders, more whitespace
- `@media print` rules: hide nav bars, profile selector, print button. Full-width. White background.

### Files

- Create: `src/routes/Report.tsx`
- Create: `src/routes/Report.css`
- Modify: `src/components/Layout.tsx` (add Report to nav)
- Modify: `src/App.tsx` (add route)

## Feature 2: User Profiles

### Data Model

```typescript
interface UserProfile {
  id: string;
  name: string;
  createdAt: string;
  preferences: UserPreferences;
}

interface UserPreferences {
  qolWeights: QoLWeights;
  scenarioOverrides: Record<string, {
    selectedCareerPreset?: string;
    customQoLRatings?: Partial<QualityOfLifeRatings>;
    dcHomeDecision?: 'sell' | 'rent' | 'keep';
  }>;
  matrixPreset: string;
  compareSelection: string[];
}
```

### State Architecture

**Shared (top-level in AppState, same for all profiles):**
- `globalAssumptions` — financial facts (age, income, home, returns, FX, daughter age)
- Base destination data (unchanged)

**Per-profile (stored in profile, swapped on switch):**
- `qolWeights` — dimension importance weights
- Per-scenario: `selectedCareerPreset`, `customQoLRatings`, `dcHomeDecision`
- `matrixPreset` — active weight preset name
- `compareSelection` — which destinations are compared

**AppState changes:**
```typescript
interface AppState {
  version: number;              // bump to 3
  globalAssumptions: GlobalAssumptions;
  profiles: Record<string, UserProfile>;
  activeProfileId: string;
  lastVisited: string;
  // REMOVED: qolWeights, scenarios, matrixPreset, compareSelection
  // These now live inside the active profile
}
```

### Derived State

All existing hooks (`useQoLWeights`, `useScenario`, `useCompareSelection`) continue to work — they just read from `state.profiles[state.activeProfileId].preferences` instead of `state` directly. The hook interface doesn't change; only the internal read path changes.

### Migration (v2 → v3)

On load, if `state.version === 2`:
1. Create a "Mekoce" profile using existing `qolWeights`, `scenarios` (extract per-scenario overrides), `matrixPreset`, `compareSelection`
2. Create a "Kara" profile cloned from Mekoce's preferences
3. Set `activeProfileId` to Mekoce's profile
4. Remove old top-level fields
5. Set version to 3

### UI — Profile Switcher

**Location**: Top bar, right side (between title and nav links on desktop; above bottom bar on mobile).

**Appearance**: Small pill/dropdown showing current profile name. Click to expand:
- List of all profiles (clicking one switches immediately)
- "Add Profile" button at bottom
- Each profile has a small "..." menu with Rename and Delete (delete disabled if only one profile)

**Behavior**:
- Switching profiles swaps the preference layer. All pages recalculate instantly.
- "Add Profile" prompts for a name (text input in a small inline form or modal), clones the current profile's preferences as starting point.
- Active profile indicated with a subtle highlight or checkmark.

### UI — Profile Management

Kept minimal:
- Add: name prompt, clone current preferences
- Rename: inline edit on the name
- Delete: confirmation prompt, disabled if it's the last profile

No dedicated settings page for profiles. The switcher dropdown handles everything.

### Report Integration

The report page header shows "Report for [active profile name]". The profile switcher on the report page lets you flip between perspectives without leaving the page. Different profiles may produce different rankings because of different QoL weights and career preset selections.

## Implementation Order

1. **Type changes**: Add UserProfile, UserPreferences interfaces. Update AppState.
2. **State migration v2→v3**: Create profiles from existing state.
3. **Refactor hooks**: Make useQoLWeights, useScenario, useCompareSelection read from active profile.
4. **Profile switcher component**: Dropdown with add/rename/delete.
5. **Wire switcher into Layout**: Top bar placement.
6. **Report page**: New route with full content.
7. **Report print styles**: @media print rules.
8. **Deploy**.

---

## Decisions from brainstorming

- Report page is a dedicated route, not a mode — clean, scrollable, print-friendly
- User profiles with flexible naming (not hardcoded to Mekoce/Kara)
- Shared: global financial assumptions. Per-profile: QoL weights, career selections, QoL overrides, DC home decision, matrix preset, compare selections.
- Profile switcher in top bar as a dropdown
- Default profiles: "Mekoce" (from existing state) and "Kara" (cloned)
- Either user can generate reports for the other by switching profiles on the report page
- State migration v2→v3 preserves all existing settings
